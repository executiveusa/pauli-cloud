# ZTE-20260716-0002 — Enterprise Runtime Plan

## Objective

Move Pauli Cloud from a bootstrap CLI to a production-grade, enterprise-ready, model-agnostic execution governance platform with enforceable policies, resumable phase state, prompt intelligence, auditable checkpoints, fleet registry, service API, CI/security gates, and reversible installation.

## Current truth

- Phase 0 bootstrap and Phase 1 capability negotiation are merged to `main`.
- The CLI is zero-dependency Node 20 and currently monolithic.
- Prompt hash verification and ICM bootstrap exist.
- Policy compilation, enforcement, checkpoint automation, prompt evaluation, fleet service, production packaging, and operational hardening remain incomplete.
- Open issue #3 defines Phase 2 policy compiler acceptance criteria.

## Target phases

1. **Policy compiler and enforcement** — compile canonical policy into Claude Code, Codex, and generic controls; preserve user files; generate restore manifest; enforce protected branches, secrets, destructive commands, and phase gates.
2. **Resumable execution runtime** — phase/bead state machine, append-only event ledger, approval registry, checkpoint and remote-SHA verification.
3. **Prompt intelligence** — immutable versions, run/evaluation records, measured promotion, learnings and failure patterns.
4. **Fleet and service mode** — multi-repository registry, authenticated local API, health/readiness/metrics, notification outbox, daily improvement report.
5. **Production hardening** — schemas, test expansion, security workflow, package validation, Docker image, Compose deployment, runbooks, threat model, support policy, and release evidence.

## Binary validation

- all commands expose `--json` machine-readable output;
- generated controls are idempotent and reversible;
- user-authored files are never silently overwritten;
- protected-branch, secret, destructive-command, and missing-phase-gate tests pass;
- phase state transitions reject invalid order and write append-only audit events;
- checkpoint verifies local and remote SHA when remote access exists;
- prompt tampering and unmeasured promotion are rejected;
- fleet API is authenticated when exposed beyond loopback and emits no secrets;
- health/readiness/metrics endpoints pass smoke tests;
- minimum 80% line coverage on critical policy, state, prompt, and API modules;
- `npm run check`, package dry-run, Docker build, dependency review, CodeQL, and secret scan pass;
- all ICM stages and PR phase ledger agree with implementation truth.

## Security

- no secret values in tracked files, logs, prompt records, or API responses;
- API token is read only from `PAULI_CLOUD_API_TOKEN`;
- default service bind is `127.0.0.1`;
- destructive and irreversible actions require explicit approval records;
- generated hooks fail closed for protected branches and likely secrets;
- external text is treated as untrusted data.

## Rollback

- generated installation manifest preserves prior file hashes and backups;
- `pauli-cloud uninstall` restores managed files;
- each phase is independently revertible;
- no production deployment or external data mutation occurs without explicit approval.

## Risk

`MEDIUM` — broad repository changes, but isolated to a new branch, no production deployment, no external secrets, and all generated changes are reversible.
